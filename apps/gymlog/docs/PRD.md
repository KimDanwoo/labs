# gymlog PRD

> 헬스장에서 폰 하나로 루틴을 따라가고, 세트를 기록하고, 쉬는 시간을 챙기고, 친구와 나누고, 달력으로 돌아보는 **모바일 우선 운동 동행 웹앱(PWA)**.
>
> 상태: **Draft (살아있는 문서)**. 확정은 ✅, 검토 필요는 ⏳, 미정은 `TBD`로 표기. 정해질 때마다 [결정 로그](#10-결정-로그)에 한 줄씩 남긴다.
>
> 이 문서는 "최소 버전"이 아니라 **완성형 제품 그림**을 기준으로 쓴다. 모든 기능은 1급 기능으로 다룬다. [9장 구현 순서](#9-구현-순서-제안)는 범위를 줄이는 게 아니라 만들 순서만 제안한다.

## 1. 한 줄 정의 & 문제

- **한 줄**: "오늘 뭘 할지 정해주고, 무게·횟수를 기록하고, 쉬는 시간을 알려주고, 진척과 기록을 친구와 나누는" 운동 동행 앱.
- **풀려는 문제**
  - 헬스장에서 매번 "뭘 몇 세트 했더라"를 머리/메모장으로 기억하는 불편.
  - 적정 무게·횟수 감이 없어 비효율적으로 운동.
  - 기구가 차 있거나 헬스장에 없을 때 뭘 대신할지 모름.
  - 쉬는 시간을 눈대중으로 재서 들쭉날쭉.
  - 꾸준함을 돌아볼 기록이 없고, 같이 하는 사람들과 동기부여를 주고받을 통로가 없음.

## 2. 타깃 사용자

- **주 사용자**: 헬스장에서 웨이트 하는 입문~중급자. 운동 중 한 손은 자유롭지 않고, 폰을 자주 들여다보기 어려운 환경.
- **사용 맥락**: 시끄럽고, 손에 땀/장갑, 세트 사이 1~3분의 짧은 조작 시간. → **큰 버튼, 적은 입력, 한 화면에 핵심만**이 설계 제1원칙.
- **함께 쓰는 맥락**: 친구·PT 그룹·운동 메이트끼리 같은 루틴을 공유하고 서로의 기록을 본다.

## 3. 핵심 설계 원칙

1. **입력 최소화**: 운동 중 타이핑은 적을수록 좋다. 기본값을 똑똑하게 채우고 사용자는 ± 조정만.
2. **세션은 풀스크린 흐름**: 한 번에 "지금 이 종목, 이 세트"만 크게 보여준다.
3. **방해는 정확한 타이밍에만**: 알림/진동은 쉬는 시간 종료 같은 명확한 순간에만.
4. **오프라인·로컬 우선, 서버는 동기화**: 헬스장 통신 불안정 가정. 로컬에 먼저 쓰고 서버와 동기화한다(소셜·다기기).
5. **기록은 자산**: 모든 세트는 남는다. 진척·공유·추천의 재료가 된다.

## 4. 횟수·무게 처리 (이 앱의 핵심) ✅

> 본인이 "관건"이라 짚은 부분. 두 축으로 설계한다 — **(A) 똑똑한 기본값 + 빠른 보정** 흐름을 기본으로 하고, **(B) 자동 카운트**를 보조 입력 수단으로 함께 제공한다. 둘은 배타가 아니라 같은 기록 모델로 합류한다.

### 4.1 똑똑한 기본값 + 빠른 보정 (기본 흐름) ✅

- **적정 횟수를 의미 있는 기본값으로 제공**: 목표(근력/근비대/근지구력/파워)×난이도별 권장 세트·reps·휴식을 `REP_SCHEMES` 테이블로 갖고 시작값을 채운다(예: 근비대·중급 4×8–12, 휴식 90s). 일반 트레이닝 가이드라인(ACSM 2026·NSCA 포지션) 기반의 **제안값이지 의학적 처방이 아니다** → UI에 면책 톤 필요. 구현: [`src/shared/training/rep-schemes.ts`](../src/shared/training/rep-schemes.ts).
- **시작 시 가감**: 세션/세트 시작 시 목표 세트·reps·무게를 ± 로 조정.
- **무게 진행(시작 무게 → 세트별 증량/유지)**: 첫 세트는 **시작 무게**(직전 기록 또는 입력)로 연다. **2세트부터 매 세트 시작 시 [증량 / 유지]를 선택**한다. 증량을 고르면 기구별 기본 증량 폭을 제안(바벨 2.5kg·머신 5kg 등, 조정 가능)하고, 유지면 그대로. 맨몸은 무게 대신 reps로 진행. 구현: [`entities/exercise/model/constants/weight-increments.ts`](../src/entities/exercise/model/constants/weight-increments.ts).
- **세트마다 실제값 기록**: 세트 종료 시 실제 reps·무게 수정 가능. 기본은 (직전 세트 무게 ± 선택분이) 채워져 있고 다르면 탭으로 조정.
- **중도 포기 허용**: 목표를 다 못 채우고 끝내면 **진행한 reps만** 기록(부분 완료). 세트를 아예 건너뛰면 skipped.
- **이전 기록 기반 자동 제안**: 같은 종목의 직전 세션 무게·reps를 다음 시작값으로 끌어와 입력을 더 줄인다(점진적 과부하 힌트 포함).

### 4.2 자동 카운트 (보조 입력) — 정식 기능

- 폰을 손에 들지 않고도 reps를 세는 보조 수단. 정확도가 100%일 필요는 없고 **"대략 채우고 끝에 한 번 보정"**이 목표.
- 후보 방식(택1 또는 병행, 정확도·기기 제약은 [8장](#8-열린-질문)에서 검증):
  - **볼륨버튼/화면 큰 탭 카운터**: 한 rep마다 한 번. 가장 단순·확실.
  - **음성 카운트**: "하나, 둘…" 혹은 호흡음 감지.
  - **모션 센서**: 기기 가속도계로 반복 패턴 감지(벤치 옆 거치/주머니).
- 어떤 방식이든 결과는 4.1과 **같은 SetLog로 합류** → 끝에 ± 보정 가능. 자동 카운트는 켜고 끌 수 있다.

### 4.3 추천 & 데이터 수집 (콜드 스타트 → 크라우드) ✅

> "사람들이 어떤 루틴으로 운동하는지"를 모아 추천의 근거로 쓴다. 두 단계로 자란다.

- **1단계 — 시드로 부트스트랩(지금 확보)**: 신규 사용자에게도 즉시 추천할 수 있도록 종목 마스터 + 대중적 분할 프리셋을 시드 데이터로 내장한다.
  - 종목 50개(부위·기구·대체관계): [`entities/exercise/model/constants/exercises.ts`](../src/entities/exercise/model/constants/exercises.ts)
  - 프리셋 루틴 12개(무분할 A/B · 상하체 · PPL · 브로 스플릿): [`entities/routine/model/constants/preset-routines.ts`](../src/entities/routine/model/constants/preset-routines.ts)
  - 목표×난이도 렙 스킴: [`shared/training/rep-schemes.ts`](../src/shared/training/rep-schemes.ts)
- **2단계 — 사용자 입력으로 보강**: 사용자가 자기 루틴을 **최소 1회 등록**(E1-1)하면, 그 루틴·실제 수행 기록이 데이터로 쌓인다. 익명·집계 기준으로:
  - **부위별 인기 종목**: "가슴 하면 보통 이 종목들" → 루틴 구성 추천.
  - **종목 동반 빈도**: 함께 묶이는 종목 → "이 루틴에 이걸 추가하시겠어요?".
  - **개인 이력 기반**: 직전 수행 무게·reps → 다음 시작값 자동 제안(점진적 과부하).
  - 그룹(E4) 안에서는 멤버들의 루틴이 그룹 추천 풀이 된다.
- **콜드 스타트 해소 순서**: 처음엔 시드 100%, 본인 기록이 쌓이면 개인 이력 우선, 모수가 모이면 집계 추천 가중. (집계 추천의 프라이버시·동의 모델은 [8장](#8-열린-질문))

## 5. 기능 범위 (Epic)

### E1. 루틴 (Routine)

- **E1-1 루틴 등록/편집**: 종목을 골라 나만의 루틴 생성/수정/삭제. 종목별 목표 세트·reps·무게(선택)·휴식시간 지정.
- **E1-2 우선순위 정렬**: 루틴 내 종목 순서를 드래그로 지정. 세션은 이 순서를 추천 순서로 따른다.
- **E1-3 프리셋 루틴(난이도·부위·목표별)**: "초급 가슴", "중급 등", "하체 상급" 등 미리 만든 루틴 카탈로그 → 복제해서 커스텀.
- **E1-4 커스텀(즉석) 시작**: 등록 루틴 없이 그 자리에서 부위/종목만 골라 바로 시작.

### E2. 운동 세션 실행 (Session) — _제품의 심장_

- **E2-1 종목 목록 출력**: 선택한 루틴/커스텀의 해야 할 종목을 횟수·세트와 함께 우선순위 순으로 표시.
- **E2-2 시작 게이팅**: 종목 선택 후 바로 시작하지 않고 회원이 기구를 찾을 때까지 대기. **"시작" 누르면 그 종목 세션 시작.**
- **E2-3 세트 진행**: 세트 "시작/종료" 버튼. 종료 시 실제 reps·무게 입력([4장](#4-횟수무게-처리-이-앱의-핵심-) 정책, 자동 카운트 연동).
- **E2-4 자동 휴식 타이머**: 세트 종료 시 자동 카운트다운 시작. 종료 시 **알림 + 진동**. 시간은 종목/목표 기본값, 조정 가능, +15s/스킵 빠른 버튼.
- **E2-5 다음 운동 추천**: 한 종목의 모든 세트를 채우면 다음 우선순위 종목을 추천 → 루틴 끝까지 진행.
- **E2-6 대체 운동**: 기구가 없거나 차 있을 때 같은 부위 대체 종목 제안 → 교체하고 계속(원종목 substitutedFrom 기록).

### E3. 기록 & 돌아보기 (History)

- **E3-1 세션 요약**: 종료 시 총 세트 수, 종목별 구성(reps·무게), 총 볼륨, 총 휴식 시간, 총 소요 시간 요약 카드.
- **E3-2 달력**: 날짜별로 어떤 루틴을 했는지 캘린더에서 확인. 날짜 탭 → 그날 세션 상세. 연속 운동(streak) 표시.
- **E3-3 진척 추세**: 종목별 무게/볼륨/1RM 추정 추이 그래프. 부위별 주간 볼륨.

### E4. 소셜 (Social) — _1급 기능_

- **E4-1 공유**: 세션 요약/루틴을 친구에게 공유(딥링크 또는 이미지 카드).
- **E4-2 그룹**: 그룹을 만들어 멤버끼리 루틴·운동 정보·기록을 공유. 권한(소유자/멤버), 그룹 루틴 라이브러리.
- **E4-3 피드/반응**: 그룹 멤버의 세션을 피드로 보고 좋아요·응원. 동기부여 루프.

### E5. 플랫폼 (PWA Platform) — _기능을 떠받치는 토대_

- **E5-1 설치·오프라인**: 홈 화면 설치, 오프라인 동작(서비스워커), 로컬 우선 저장 + 서버 동기화.
- **E5-2 진동·알림**: `navigator.vibrate`, Notification API(휴식 종료·세션 리마인드). 기기별 제약은 [8장](#8-열린-질문).
- **E5-3 화면 유지**: 세션 중 Screen Wake Lock으로 화면 꺼짐 방지.
- **E5-4 계정·동기화**: 다기기·소셜을 위한 인증. 로컬-only 게스트 → 계정 연동 업그레이드.

## 6. 데이터 모델 (초안, `TBD` 확정 전)

> 시드 부분(Exercise·Routine·RepScheme)은 **구현 완료**. 세션·소셜은 초안.
> FSD cross-import 금지 때문에 두 엔티티가 공유하는 어휘(Goal·Difficulty·RepScheme)는 `shared/training`에 둔다.

```text
Exercise(종목)  ✅ entities/exercise
  id, nameKo, nameEn, primaryMuscle(부위), secondaryMuscles[],
  equipment(기구), movementType(compound|isolation),
  alternativeIds[]                 // 대체 운동(같은 부위)

RepScheme  ✅ shared/training         // 적정 기본값은 종목별이 아니라 목표×난이도 테이블에서 파생
  REP_SCHEMES[goal][difficulty] = { sets, repsMin, repsMax, restSec, loadPctMin, loadPctMax }

WeightIncrement  ✅ entities/exercise   // "증량" 선택 시 제안 폭(kg), 기구별
  WEIGHT_INCREMENT_KG[equipment] = number   // 바벨 2.5 · 머신 5 · 덤벨 2 · 맨몸 0

Routine  ✅ entities/routine
  id, name, split, dayLabel, goal, difficulty, source(preset|custom|group)
  items: [{ exerciseId, order(우선순위), targetSets?, targetReps?, targetWeight?, restSec? }]
         // target* 비우면 goal×difficulty의 REP_SCHEMES 기본값을 상위 레이어에서 해석

WorkoutSession
  id, routineId|null(custom), userId, date, status(active|done|aborted)
  startedAt, endedAt, totalRestSec
  performed: [ExercisePerformance]

ExercisePerformance
  exerciseId, order, status(done|skipped|substituted), substitutedFrom?
  sets: [SetLog]

SetLog
  index, targetReps, actualReps, weight,
  status(done|partial|skipped), restSec,
  inputSource(manual|auto-button|auto-voice|auto-motion)

Group / Membership / Reaction        // 소셜
  Group: id, name, ownerId, routineIds[]
  Membership: groupId, userId, role(owner|member)
  Reaction: sessionId, userId, type(like|cheer)
```

- 저장: **로컬(IndexedDB) 우선 + 서버 동기화**. 서버 스택은 `TBD`(Supabase 후보 — 인증·DB·실시간 한 번에).

## 8. 열린 질문 (채워야 할 것)

- ✅/⏳ **적정 기본값의 근거**: 1차로 ACSM 2026·NSCA 포지션 기반 `REP_SCHEMES` 테이블 확보. 남은 것 — 사용자 1RM 입력 기반 무게 자동계산 도입 여부, UI 면책 문구 문안.
- ⏳ **집계 추천의 프라이버시·동의**: 사용자 루틴/기록을 익명·집계로 추천에 쓸 때의 동의 범위·옵트아웃.
- ⏳ **자동 카운트 정확도·방식**: 볼륨버튼/음성/모션 중 실제로 쓸 만한 것 검증(기기·브라우저 권한 제약 포함).
- ⏳ **PWA 알림·진동 한계**: iOS Safari 백그라운드/화면 꺼짐 상태 제약 검증 — 휴식 종료 알림이 핵심이라 선검증.
- ⏳ **서버 스택·동기화 모델**: Supabase? 충돌 해소(로컬↔서버) 전략.
- ⏳ **종목 마스터 데이터 규모·출처**: 부위/기구/대체관계 데이터 어디서 확보.
- ⏳ **인증·프라이버시**: 게스트→계정 전환 시점, 기록 공개 범위 기본값.

## 9. 구현 순서 제안

> 범위 축소가 아니라 **만들 순서**. 각 묶음은 그 자체로 쓸 수 있는 가치를 준다.

1. **세션 흐름 골격**: E2 전체(시작 게이팅·세트 진행·휴식 타이머+진동·다음 운동) + E1-1/E1-2 루틴 등록·정렬 + 4.1 기본값 흐름 + E3-1 요약. (혼자 운동 따라가기 완성)
2. **기록·돌아보기**: E3-2 달력, E3-3 추세, 이전 기록 기반 제안, E5 플랫폼(설치·오프라인·화면유지) 다지기.
3. **카탈로그·대체**: E1-3 프리셋 루틴, E2-6 대체 운동, 종목 마스터 데이터 확충.
4. **함께**: E5-4 계정·동기화 → E4 소셜(공유 → 그룹 → 피드).
5. **자동 카운트**: 4.2 검증 후 가장 정확한 방식부터 옵트인 제공.
